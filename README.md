# TicketFlow

A self-hosted event ticketing platform — an MVP inspired by Booked It. Sell
tickets for your own events, issue scannable QR tickets, take payments, and
check people in at the door.

It runs on **plain Python 3 with no dependencies to install** — no `pip install`,
no Node, no database server. Just Python's standard library and a single SQLite
file.

## Features

- **Public event listings** with cover accents, dates, and "from" pricing.
- **Checkout** with a quantity picker, live total, and buyer details.
- **Payments** via Stripe Checkout (test mode) — with a built-in **mock mode** so
  it runs end-to-end with zero configuration.
- **QR tickets** generated by a self-contained QR encoder (no libraries), rendered
  as crisp inline SVG. Every ticket has a unique code.
- **Door scanner** (`/scan`) that uses the phone/laptop camera to check people in,
  with clear *Admitted / Already used / Invalid* feedback — plus a manual-entry box.
- **Organiser dashboard** (`/admin`) to create and publish events, manage ticket
  types, and see sales, revenue, check-ins, and recent orders.

## Quick start

```bash
python3 run.py
```

Then open **http://localhost:8000**.

- Public site: http://localhost:8000
- Organiser dashboard: http://localhost:8000/admin  (default password: `admin123`)
- Door scanner: http://localhost:8000/scan

On first run the database is created and seeded with a few sample events so the
site looks alive. That's it — no other setup.

> Requires Python 3.8+ (macOS and Linux ship this by default). Check with
> `python3 --version`.

## Enabling real (test-mode) card payments

Out of the box the app uses **mock payments** — checkout is simulated and no card
is charged. To use real Stripe test-mode payments, grab your **test** secret key
from the [Stripe dashboard](https://dashboard.stripe.com/test/apikeys) and set it
before starting:

```bash
export STRIPE_SECRET_KEY="sk_test_..."
python3 run.py
```

Checkout will then redirect to Stripe's hosted page. Use Stripe's test card
`4242 4242 4242 4242`, any future expiry, any CVC. On return, tickets are issued
once the session is confirmed paid.

## Configuration

All optional, set as environment variables:

| Variable            | Default            | Purpose                                    |
|---------------------|--------------------|--------------------------------------------|
| `PORT`              | `8000`             | Port to listen on                          |
| `ADMIN_PASSWORD`    | `admin123`         | Organiser dashboard password               |
| `STRIPE_SECRET_KEY` | *(empty)*          | Stripe **test** secret key; empty = mock   |
| `TICKETFLOW_DB`     | `./ticketflow.db`  | SQLite database file location              |

## Scanning from a phone

The scanner uses the browser's `BarcodeDetector` and camera, which require a
**secure context**. `localhost` counts as secure on the same machine. To scan
from a phone on your network you'll need to serve over HTTPS (e.g. put it behind
a reverse proxy, or use a tunnel like `ngrok`/`cloudflared`). The manual-entry
box works anywhere as a fallback.

## Tests

```bash
python3 tests/test_app.py   # full purchase → ticket → scan → admin flow
python3 tests/test_qr.py    # QR decode verification (needs opencv+numpy)
```

`test_app.py` uses only the standard library. `test_qr.py` needs `opencv-python`
and `numpy` (`pip install opencv-python numpy`) purely as an independent decoder
to *verify* the generated QR codes — the app itself never needs them.

## How it fits together

1. A buyer picks tickets and checks out → an **order** is created (`pending`).
2. Payment is taken (Stripe or mock). On success the order is marked **paid**,
   stock is decremented, and one **ticket** (with a unique QR code) is issued per
   seat.
3. The buyer sees their QR tickets and can reopen any ticket at `/t/<code>`.
4. At the door, `/scan` reads the QR and calls `/api/scan`, which admits the
   ticket once and reports repeats as *already used*.

## Notes & next steps

This is an MVP foundation intended to be built on. Natural next steps:

- Email delivery of tickets (e.g. SMTP) — the code marks where this would go.
- A Stripe **webhook** endpoint for payment confirmation independent of the
  browser redirect (the success page already verifies the session as a fallback).
- Per-organiser accounts / multi-tenant support.
- Refunds, discount codes, reserved seating, and CSV exports.
